Critical Flaw Resolution

Shared BLOCKING Flaw: fillForm() destroys Repeater UUIDs

Filament's Repeater component regenerates internal UUIDs on every fill() call via afterStateHydrated. This destroys the user's editing context (collapsed states, cursor positions, unsaved edits). All three plans hit this.

Solution: Redirect-based Apply

When the user clicks "Apply Changes", redirect to the same page with ?apply_draft=1. The page mounts fresh, mutateFormDataBeforeFill() reads from draft instead of pages, and all Filament lifecycle hooks run normally. No fillForm() needed.
