Problem

Current behavior

All errors render identically in the input zone with a generic message. Users cannot tell if the provider is down vs a one-off failure.

Could not generate a response

Same for transient errors AND single failures

Desired behavior

When retry exhaustion indicates a systemic outage, show a distinct top-of-drawer banner and disable chat input.

Mia is temporarily unavailable

We're having trouble connecting. Please try again in a moment.

Mia is temporarily unavailable...

Key design insight

If a transient error (ConnectionException, 5xx) reaches handleStreamError(), it has already passed through executeWithRetry() which exhausted all 3 attempts. The exception type already encodes whether retries were exhausted — no wrapper exception needed.

Add isProviderUnavailable() helper

AssistantDrawer.php 
private function isProviderUnavailable(Throwable $error): bool
{
    if ($error instanceof ConnectionException) {
        return true;
    }

    if ($error instanceof RequestException
        && $error->response !== null
        && $error->response->status() >= 500
    ) {
        return true;
    }

    return false;
}

Guards $error->response !== null to prevent TypeError when RequestException lacks a response object.

private function handleStreamError(Throwable $error): void
{
    // Always clean up optimistic UI
    if (!empty($this->messages) && end($this->messages)['role'] === 'ai') {
        array_pop($this->messages);
    }
    if (!empty($this->messages) && end($this->messages)['role'] === 'user') {
        array_pop($this->messages);
    }

    $this->isStreaming = false;
    $this->activeTools = [];
    $this->followUpChips = [];  // ← NEW: prevents stale chips

    // Classify error
    if ($this->isProviderUnavailable($error)) {
        $this->errorState = $this->assistantName . ' is temporarily unavailable';
        $this->errorType = 'provider_unavailable';
    } else {
        $this->errorState = 'Could not generate a response';
        $this->errorType = 'network_error';
    }

    // Always report — unconditional (no blind spots)
    Report::toError($error);
    $this->analyticsService->track(AiAnalyticsEvent::ErrorDisplayed, [
        'error_type' => $this->errorType,
        'portal'    => $this->tier,
    ]);
}

Ambiguity: reactive vs proactive detection

AC-1.8.4 says "When the user opens the drawer" — could mean proactive health check. This plan uses reactive detection (errors surface when user sends a message). This matches AC-1.8.3's pattern and avoids adding a health-check endpoint that doesn't exist.

Design decisions

Why no sendMessage() guard? ▶

retryLastMessage() clears errorType before calling sendMessage(), and sendMessage() itself clears it at line 183-184. The guard can never fire. Alpine disabled bindings are sufficient.

Why no Dismiss button on banner? ▶

Provider outage is a systemic state. Dismissing the banner would leave the user with a disabled input and no visible explanation. The escape hatch is closing the drawer or retrying.

Why not use x-common.lumion-alert? ▶

It wraps <x-filament::section> which adds card chrome unsuited for flush drawer banners. Consistent with the existing error banner at line 431 which also doesn't use it.

Why !$isStreaming gate on banner? ▶

Prevents flicker during retry attempts. Banner hides while streaming, reappears if the second attempt also fails with a transient error.

Why no new wrapper exception? ▶

The exception type at handleStreamError already encodes whether retries were exhausted. executeWithRetry() only rethrows transient errors after all 3 attempts. No wrapper needed.

Why clear followUpChips? ▶

Without clearing, stale follow-up chips from a previous successful exchange could appear alongside the provider unavailable banner. Found in Round 3 adversarial review.
