Responsive Mobile Layout

1 GAP
✓ Full-width layout on mobile (<768px)
✓ Drag handle hidden on mobile
✓ Back button replaces close button

✓ visualViewport resize listener registered

⚠ keyboardHeight computed but never applied to input bar positioning
Gap — Medium

keyboardHeight unused: The visualViewport.resize listener computes keyboardHeight (line ~242 of assistant-drawer.blade.php) but it is never bound to any element's style. The input bar uses static env(safe-area-inset-bottom) padding. On iOS mobile, the virtual keyboard could overlap the input field if the safe-area-inset does not account for keyboard height.

Impact: On some iOS devices, the input bar may be partially obscured by the keyboard. Android typically handles this via viewport resize. Low real-world incidence but inconsistent with the computed property existing.

Evidence: assistant-drawer.blade.php — handleViewportResize() sets this.keyboardHeight but no :style binding references it. Input bar uses pb-[env(safe-area-inset-bottom)].

Accessibility Certification & Audit

2 GAPS + 2 DRIFT
✓ Audit report created at app/Domains/AI/docs/accessibility-audit-report.md
✓ axe-core: 0 violations (drawer-scoped)
✓ 27 Pest tests + 4 Dusk browser tests
✓ VoiceOver: 18/37 test cases pass
⚠ Lighthouse accessibility score not run (deferred)
⚠ axe-core not run on all 5 drawer states
⇄ No dedicated NVDA testing (Chrome a11y tree used)
⇄ VoiceOver 17 cases deferred to future components
Gap — Low

Lighthouse score not run: The story acceptance criteria expected a Lighthouse accessibility score. The audit report documents this as deferred, using axe-core as an equivalent automated tool. axe-core is the engine behind Lighthouse's accessibility audit, so coverage is functionally equivalent.

Gap — Medium

axe-core scope limited: The audit covers "drawer-scoped" axe-core (Open/Welcome state). The PRD specifies 5 drawer states: Open/Conversation, Open/Streaming, Open/Error, Mobile, and Welcome. Only Welcome was tested with axe-core. The other states may have different DOM structures that introduce violations.

Impact: Streaming and error states add dynamic DOM elements (loading indicators, error banners) that were not axe-scanned. Risk is mitigated by Pest tests asserting ARIA attributes on those elements.

Drift — Acceptable

No NVDA testing: Chrome accessibility tree inspector was used as a documented equivalent. NVDA is Windows-only; the development environment is macOS. This is an acceptable platform limitation.

Drift — Acceptable

VoiceOver 17/37 deferred: The deferred test cases relate to components not yet built (tool confirmations, history panel, quick action cards). These are Epic 3-6 features. The 18 passing cases cover all currently implemented UI.

Evidence: app/Domains/AI/docs/accessibility-audit-report.md — full audit report with axe-core results, VoiceOver matrix, and test coverage.
